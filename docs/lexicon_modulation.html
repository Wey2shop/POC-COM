Speaking the Channel

A pre-rendered vocabulary of real speech clips as POC-COM's data link — no TTS engine, no external process, no OS-specific API, and four physical layers it took to get here.

POC-COM moves structured messages and open-board posts over a Push-to-Talk-over-Cellular radio using nothing but the voice audio path — no data channel, no API, no paired app on the far end required. Every earlier version of this project tried to make that audio path carry a purpose-built signal: tones, chirps, frequency-shift-keyed symbols, and later live-synthesized speech. Every one of those either fought a voice codec (AMR/SILK-family, aggressively tuned to preserve human speech and reshape or discard anything that doesn't fit that model) or turned out to depend on something that couldn't ship the way the project needed to ship — a live TTS engine tied to one operating system. This document is the record of that whole progression, why each stage ended, and the design that finally stuck.

The current design, in one pass

A message's fields are spelled out as a sequence of tokens — NATO phonetic letters, digit words, and the field's own marker word — and each token maps to a short, real audio clip that was recorded exactly once, offline, and embedded in the binary. Sending a message is pure sample-buffer concatenation: 

READY FROM K J 4 A B C RECIPIENT B A S E LOCATION I O 8 0 SUBJECT S T A T U S MESSAGE A L L SPACE Q U I E T COMPLETE

each token's pre-recorded clip is looked up (clips.rs) and appended to the output buffer with a fixed, exact gap of digital silence in between (modulator.rs's assemble_from_tokens) — no TTS engine runs at send time, on either platform, at all. On the receive side, a real Whisper base.en speech-recognition model (running via candle, a pure-Rust ML framework — no Python, no network call, the whole model is embedded in the binary) transcribes whatever comes out of the radio's speaker. Decoding is grammar-constrained (see Phase 8 below): at every generation step, the model's own logits are masked down to only the tokens that keep the output a legal continuation of a known vocabulary word, so nothing outside this crate's own ~49-word list can ever come out, structurally. The transcript is searched for ready and complete as whole words, and the text between each pair of field markers (from/recipient/location/ subject/message for Mail; writer/post/link/grid for Social) becomes that field's value, reconstructed letter-by-letter (phonetic.rs's unspell).

Every field is spelled this way now — compact identifiers (callsigns, Maidenhead locators) and free text (Subject/Message/Post) alike — since there's no live TTS engine left to speak arbitrary natural-language sentences directly. A word boundary inside free text is itself a spoken token (SPACE), since there's no way to recover "where one word ends and the next begins" from transcript text alone without one. Social's link field (a URL suffix) also spells a literal . as its own token (DOT), since a file extension matters for reconstructing a valid link.

Why the previous (live-TTS) design got replaced: it worked, but it was Windows-only (the WinRT/SAPI speech APIs it called don't exist on Linux), and this project needs a real Linux release build too. Pre-rendering the entire vocabulary once, offline, and shipping the resulting clips as embedded data sidesteps the problem entirely: clip playback is pure Rust with zero OS dependency, so it's identical on both platforms. Only clip generation (dev_tts.rs, gated behind a Windows-only, opt-in gen-clips Cargo feature that never ships in the app) still touches a TTS engine, and only once, by a developer, not at runtime.

This also turned out to fix a standing reliability question for free: since every clip is generated once and shipped as bit-identical data, every empirical finding this project makes about which words and pauses are reliable stays true forever, on every install — it's no longer at the mercy of whichever voice a given machine's TTS engine happens to default to (a real, confirmed problem with the previous design: switching the default voice changed which words got misheard).

Pause duration between clips is now a literal, exact number (gap_ms) rather than an emergent property of punctuation fed to a live TTS engine. It used to be a user-facing Slow/Normal/Fast picker (250/150/100ms); real live-hardware testing — not just this app's own offline round-trip tests, which pipe assembled audio straight into Whisper and never touch a real speaker, microphone, PTT radio, or VAD at all — surfaced failures at the fastest setting, so the picker was removed in favor of one fixed, more conservative gap (200ms). Widening the gap isn't a cure-all, though — one real failure (below, "Reddy, Ryder") turned out to be completely unaffected by widening the gap from 60ms to 100ms, meaning it wasn't a blending problem at all. A separate 300ms of silence is also now prepended before the very first clip of every message — see Phase 6 below.

Four tries to get here

Each earlier design is described below roughly in the order it was tried, together with the specific, concrete finding that ended it. None of these were abandoned on a hunch — each one worked well enough to get real end-to-end tests passing before the next problem showed up.

Attempt 1 — tones: chirps, tone-chords, then multi-lane FSK

The channel bandwidth was split into narrow frequency lanes, each continuously transmitting one of a small set of tones or cyclically-shifted chirps — first as Helical Spread Modulation (a pilot chirp lane for continuous timing/drift correction plus data lanes), later as on/off tone-chord keying, and finally as a 3-lane FSK scheme with fountain-coded (RaptorQ) redundancy. Each iteration got real-hardware validation and real bug fixes (a coarse-to-fine sync search that let the winning candidate register as its own runner-up and collapse its confidence margin; a real ~18-second periodic burst erasure found on live PTT hardware, fixed by adding redundancy).

The fundamental problem never went away: a codec built to preserve speech actively reshapes or discards anything that doesn't look like speech. Every retune bought some more headroom, but the whole approach was spending its effort fighting the channel's own priorities rather than working with them.

Attempt 2 — a closed dictionary of isolated TTS words

The insight that flipped the approach: a voice codec passes real speech through essentially perfectly, because that is exactly the signal it is built to preserve. So instead of a signal shaped to survive being misclassified as noise, why not use a signal built to be classified as speech in the first place? The first version of that idea pre-rendered a closed vocabulary of ~2048 individual English words via TTS, encoded arbitrary bytes as sequences of those words (each word standing in for a fixed bit pattern), and decoded with Whisper doing per-word closed-set classification.

This traded one fight for another. Whisper is a language model as much as an acoustic one, and a sequence of grammatically unrelated isolated words triggered its own "correction" instinct — it would confidently transcribe eight unrelated dictionary words as a fluent-sounding phrase sharing almost none of the actual words. Constraining decoding at every generation step to only legal dictionary-word continuations (a per-step token trie) fixed that specific failure mode completely. But even with hallucination eliminated, per-word recognition accuracy on an artificial closed vocabulary — compounded by weak, phonetically-reduced function words (the → they) sorting to the front of a frequency-ordered word list and landing on the most bit-significant codes — wasn't reliable enough even for messages a few words long. A real test with a compressed 6-byte payload failed its checksum non-deterministically depending on exactly which dictionary words that particular byte pattern happened to map to.

Attempt 3 — live TTS, real sentences, free-form decode

The fix was to stop fighting both systems at once. A voice codec passes real speech through cleanly. Whisper transcribes real, fluent speech accurately — that's the one thing it is unambiguously built and trained to do well. Isolated artificial word sequences are the thing that breaks both assumptions at once: they're speech-shaped enough to survive the codec, but not real language, which is exactly what trips Whisper's own correction instinct. Real natural sentences (or, once found unreliable on bare alphanumeric identifiers, NATO-phonetic-spelled ones) satisfy both systems simultaneously, so there's no adversarial relationship left to manage. Synthesis first shelled out to a PowerShell/SAPI subprocess per message, then moved in-process via the WinRT Speech Synthesis API (windows-rs COM/WinRT bindings, no subprocess, no temp files).

This worked — genuinely, with real end-to-end tests passing — but it was Windows-only. WinRT and classic SAPI don't exist on Linux, and this project needs a real Linux release build. Live synthesis also meant every empirical reliability finding was only as good as whatever voice a given machine's TTS engine defaulted to: swapping the default WinRT voice for testing changed which specific words came back misheard, so "reliable" was never a fixed, permanent property of the design — it was a property of today's voice.

Attempt 4 (current) — a small fixed vocabulary, pre-rendered once

The realization: this project doesn't actually need arbitrary live speech synthesis at all. Every message this app ever sends is built from the same small, closed set of words — 26 NATO letters, 10 digit words, a handful of field markers, and two structural separators (Space, Dot). Render that whole vocabulary to real audio exactly once, embed the result, and speaking a message becomes pure sample-buffer concatenation with no TTS engine, no OS API, and no external process involved at send time on either platform — see the design above.

What this bought: no live TTS dependency of any kind at runtime (so a real Linux build is possible at all), bit-identical audio on every install forever, and an exact, sweepable inter-clip gap instead of punctuation-dependent guesswork. What it cost: Subject/ Message/Post now pay the same per-character price identifiers always did — every character is a whole spoken clip, not a fraction of a natural sentence — so free-text field limits are markedly shorter than the live-TTS era's.

Building this out for real (not just in principle) surfaced three more concrete findings, the same "stop guessing, test with real audio" method this whole project has used throughout:

  • "Papa" (NATO's own word for P) came back transcribed as "Harper" in a real end-to-end test — dropping the letter silently, since neither word matched. Rather than add another homophone-tolerance entry, the vocabulary just uses a different, distinctly-sounding word for P: Panda.
  • The Social "author" marker went through three names before one stuck. "Author" came back as "Offer"; the next attempt, "Sender", fared no better ("Sendre", and split into "Send a"); "Writer" has held up since, and is phonetically distant from every other word in the vocabulary (deliberately, so it can't get acoustically confused with "Post").
  • Not every mismatch is a mishearing — some are just an alternate spelling of the same word Whisper heard correctly. A spoken "Yankee" has come back written as both "yanki" and, in a separate real test, "yankie" — two different informal spellings of a word Whisper actually recognized correctly. phonetic.rs's decoder tolerates both the digit-homophone case (a different word, e.g. "for" for "4") and this alternate-spelling case (the same word, differently spelled) with the same small lookup table.

One Fast-speed test failure ("Reddy, Ryder" for "Ready, Writer" ahead of one specific callsign) turned out to be neither of the above: widening the gap from 60ms to 100ms produced the exact same wrong transcript, ruling out blending entirely. This looks like Whisper's own cross-attention re-interpreting the whole utterance based on content later in the audio — a real, if rare, quirk of the model's architecture rather than something a word swap or gap change fixes; a different callsign in the same slot passes reliably.

Phase 6 — what real, non-synthetic usage found

Every finding above came from this crate's own round-trip tests: assembled audio piped straight into Whisper, with no real speaker, microphone, PTT radio, or this app's own voice-activity detection anywhere in the loop. Actually running the app against a live radio path surfaced three more problems that no synthetic test could have caught:

  • The wake word "Ready" came back transcribed as "Reddy", breaking the whole message's parse before it even reached a single field. This project hit the identical failure mode once before, in the now-deleted fsk_modem layer: whatever sits at sample 0 of a transmission is uniquely exposed to receiver-side squelch/AGC/VAD settling and mic/speaker onset transients in a way nothing later in the audio is. The fix is the same one that worked there — 300ms of real silence prepended before the whole message (modulator.rs's LEAD_IN_MS), giving the receive chain time to settle before anything acoustically meaningful starts.
  • A missing grid marker let post silently absorb garbled data instead of failing the parse. link and grid used to be spoken only when the field was actually present, which made a genuinely absent field indistinguishable from a misheard marker word — and when grid was misheard, post's value silently extended all the way to complete, swallowing whatever garbled fragments of the grid value came after it. Two real "partial decode" transcripts showed exactly this: a post field visibly containing fused fragments of what should have been a separate grid value. The fix: link and grid are now always spoken markers, even when the field is empty (spelling an empty string produces no clip tokens at all, just the marker word itself). A transcript now missing either marker is unambiguously a transcription failure, so the whole parse fails rather than returning corrupted data — the same "parse succeeds or parse fails, no partial credit" rule this format already applied everywhere else.
  • The Slow/Normal/Fast gap picker was collapsed to one fixed value. Both real failures above were reported at the fastest (100ms) setting; rather than keep three variants to reason about, the picker was removed entirely in favor of one more conservative fixed gap (200ms) — one less axis of variation when a real transmission gets misheard.

Two more real reports arrived immediately after the fixes above shipped, both still on real PTT hardware:

  • "Panda" (P's word, already a replacement for the unreliable NATO "Papa") failed two more ways — transcribed as "Peter" in one real transmission, and dropped without a trace in another (an author callsign lost its leading letter; a link's ".png" extension lost its "P" and read back as ".ng"). Two different wrong outcomes for the same clip isn't a stable alternate spelling the way "Yankee"/"yanki" was — it means the word itself is unreliable, not just occasionally misheard, so the same fix as before applied again: swap it. P now uses "Popcorn", a longer, more phonetically distinct word.
  • "Writer" (the AUTHOR marker) came back transcribed as "right", breaking the whole parse. Rather than swap the word a fourth time — regenerating the clip library isn't free, and (per the finding below) isn't even perfectly reproducible — marker words got the same kind of tolerance letters/digits already had (phonetic::HOMOPHONES): a small MARKER_ALIASES table in message.rs that accepts known alternate transcriptions of a marker word without needing new audio at all.

Also confirmed the same day: a mishearing report that turned out not to need new audio at all can still cost you a full clip-library regeneration if you change even one unrelated word. Fixing "Popcorn" meant re-rendering all 49 clips from scratch (this crate's generator always renders the whole vocabulary, not just the changed entry) — every clip is fresh TTS output, not just P's. The full real round-trip suite was re-run against the new clips.bin before trusting it, per this crate's own standing rule that clip generation isn't bit-for-bit reproducible run to run.

Phase 8 — constraining decode to this crate's own vocabulary

Every fix through Phase 6/7 treated a specific mishearing after the fact: tolerate a known alternate transcription, or swap the underlying word. But free-form Whisper decode can, in principle, output any English word for a degraded clip — "Writer" → "right", "Panda" → "Peter", the wake word "Ready" → "Ita" in one report — and there's no way to fully enumerate every wrong word a real, noisy radio path might eventually provoke. Chasing each one individually is unbounded.

decoder.rs's decode loop was already hand-written token-by-token (a manual greedy argmax over the model's logits at each step, not a canned high-level "transcribe" call) rather than using a library's built-in generation function — which meant the exact hook needed to fix this at the root was already there. The loop now builds a small trie, once at load time, from the BPE tokenization of every vocabulary word (both title-cased and lowercase, since Whisper's own casing choice for a given word isn't consistent) via WordGrammar, and at every generation step masks the model's logits down to only the tokens that keep the output a legal continuation of some known word. The model is structurally unable to emit "Ita" or "right" or any other out-of-vocabulary word ever again — it can still pick the wrong known word, but never an unknown one.

Why this isn't Attempt 2 again: this project already tried constrained decoding once, in the abandoned 2048-word bit-packed dictionary scheme, and that had real, separate problems — weak, phonetically-reduced function words landing on bit-significant codes, so even with hallucination eliminated, per-word accuracy still broke a payload's checksum. Nothing here is bit-packed and the vocabulary is ~25x smaller (49 words vs. 2048), so one wrong word costs one character, not a whole message. The technique that fixed hallucination there is being reused here for a different, much cheaper-to-tolerate problem.

A new test, transcript_never_contains_a_word_outside_the_vocabulary, asserts exactly that property directly against a real decode, not just indirectly through parsing success. Every existing real round-trip test (Mail, Social with/without optional fields, the Reddy/Ryder regression case) stayed green against the new decode path with no changes needed elsewhere — the fix lives entirely inside decoder.rs.

Design principles that carried through every rewrite

  • Voice-channel only. No data-channel or API assumption on the far end — if a radio can carry a voice call, it can carry POC-COM traffic.
  • No volume controls, no passthrough. Structural, not policy: there's no code path anywhere that reads an input stream and writes it to an output stream.
  • Empirical over theoretical. Every parameter this project has ever tuned — band edges, symbol duration, redundancy overhead, inter-clip gap, even which word to use for a given letter — was tuned against direct measurement (first simulated, later real hardware and real models), not derived purely from theory.
  • Stop fighting the channel; become the signal it's built to carry. The throughline across every PHY attempt, and the whole reason this document exists.
  • Ship on every platform you promise, not just the one that was easiest to build on first. The reason Attempt 3 didn't survive despite working — a real Linux build isn't optional, so a Windows-only runtime dependency wasn't acceptable no matter how well it tested.

Honest current limits

  • No forward error correction of any kind. A misheard marker word breaks that field's parse (or the whole message's, if a marker itself is missed). Grammar-constrained decode (Phase 8) eliminated the out-of-vocabulary class of mishearing structurally, but a degraded clip can still be decoded as the wrong known word (e.g. one letter confused for an acoustically similar one) — that residual risk is what the marker-word framing and per-word tolerance tables still exist to catch.
  • Every character of every field now costs one whole spoken clip plus a gap — there's no natural-sentence speaking rate to lean on anymore, hence the deliberately short per-field character caps in pipeline.rs (12/12/24/60/60 for From/To/Subject/Message/Post).
  • Real over-the-air PTT testing has now happened (see Phase 6) and found real problems this crate's own offline round-trip tests structurally could not — onset clipping on the wake word, and a marker-absence ambiguity in Social's grid/link fields. Both are fixed and covered by new real round-trip tests, but the same discipline applies going forward: every future finding should come from testing against a real radio and microphone, not just an assembled-audio→Whisper pipe. The Linux build has now actually been compiled and run on a real Linux machine (WSL Ubuntu) — the full workspace test suite, including every real end-to-end round trip, passes natively there, confirming the design intent (no `windows` crate anywhere in the default feature set) actually holds up in practice, not just in theory.
  • The link field (a Social attachment's shortened URL suffix) and the free-text fields (Subject/Message/Post) get the same spell/unspell treatment as identifiers, but have had less adversarial real-world testing than From/To/Location/Grid specifically — the three real findings above all surfaced through identifier and marker-word testing.
  • Clip generation itself isn't perfectly reproducible run-to-run (confirmed: regenerating the full vocabulary after an unrelated word change shifted which other, untouched words transcribed reliably) — each checked-in assets/clips.bin needs the real round-trip test suite run against it specifically before being trusted, not assumed identical to a previous generation.
